# PRD - AI 招投标助手

## 一、产品定位

本项目是企业内部使用的“辅助投标工作台”。

目标不是做营销官网，也不是做泛 AI 演示平台，而是围绕招投标日常工作提供一个可运行、可验证、可逐步扩展的 MVP 系统。

当前核心目标：

1. 上传招标文件
2. 解析招标文件
3. 抽取核心字段
4. 判断是否建议投标
5. 生成标书初稿
6. 接入简易企业知识库

## 二、目标用户

- 投标专员
- 商务人员
- 项目经理
- 售前支持人员

## 三、当前要解决的问题

- 招标文件阅读和信息整理耗时
- 核心字段抽取依赖人工
- 投标判断依赖个人经验，标准不一致
- 标书初稿编写重复劳动多
- 企业已有案例、资质、模板难以复用

## 四、当前 MVP 范围

### 1. 招标主链路

用户上传招标文件后，系统需要完成：

1. 保存文件
2. 解析文本
3. 抽取结构化字段
4. 输出投标建议
5. 输出标书初稿

当前支持情况：

- 优先支持 `txt / docx`
- `pdf` 当前仅预留接口和上传入口，解析未完成

### 2. 简易企业知识库

知识库当前只做简易版，不做复杂 RAG。

支持资料类型：

- `company_profile`
- `qualifications`
- `project_cases`
- `templates`

支持能力：

- 文档上传
- 分类管理
- 文本解析
- 规则切块
- 简单检索
- 给 Agent 提供上下文
- 前端资料中心管理页（上传 / 列表 / 处理 / 检索调试）

### 3. Agent 调用链路

当前只允许以下 4 个 Agent：

1. `extract_agent`
2. `judge_agent`
3. `generate_agent`
4. `orchestrator`

当前要求：

- Agent 不直接访问数据库
- 由 backend orchestrator 先检索知识
- 将知识片段整理成上下文后再喂给 Agent
- 当前阶段允许 mock LLM 输出，但流程必须真实

## 五、关键业务规则

### 1. judge_agent 使用的知识来源

- `qualifications`
- `project_cases`

### 2. generate_agent 使用的知识来源

- `company_profile`
- `templates`
- `project_cases`

### 3. 检索规则

当前阶段不做向量检索。

检索方式固定为：

- 分类过滤
- tags 过滤
- industry 过滤
- `LIKE` 关键词匹配

### 4. 错误处理

- 所有接口必须返回统一 JSON 结构
- 所有失败必须返回可读 `message`
- 前端页面必须有 `loading / success / error / disabled` 状态

## 六、当前 UI 要求

前端定位为“企业内部辅助投标工作台”。

设计方向：

- 专业、克制、现代
- 信息层级清晰
- 适合表单、文档、结果展示
- 不做 AI 炫技风

当前实现方向：

- 左侧导航工作台布局
- 招标上传页采用双栏
- 结果页采用结论摘要 + 文档阅读区 + 证据区
- 知识库页作为资料中心管理页（上传 / 列表 / 处理 / 检索调试）
- 技术上采用 `React + Tailwind`

## 七、当前阶段非目标

- 不做复杂权限系统
- 不做审批流
- 不做自动同步企业网盘
- 不做复杂向量检索
- 不做 rerank
- 不做全网自动爬虫
- 不做高保真标书排版导出

## 八、当前成功标准

- 用户能够完成一次从上传到生成的完整链路
- `judge` 和 `generate` 能够使用知识库片段
- 页面能清晰展示流程状态与结果
- 文档、接口和当前实现保持一致
